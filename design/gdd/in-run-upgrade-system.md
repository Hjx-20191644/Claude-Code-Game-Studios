# 局内升级系统 (In-Run Upgrade System)

> **Status**: Approved
> **Author**: user + agents
> **Last Updated**: 2026-04-27
> **Implements Pillar**: 每一秒都在做选择、小而深

## Overview

局内升级系统是"战斗-升级"循环中的喘息与抉择时刻。当波次系统发出升级请求后，本系统接管游戏流程：从升级池抽取 3 个升级选项，展示给玩家，等待选择，将选择结果回传升级池并通知各玩法系统应用效果。本系统不定义升级内容（由升级池系统驱动），不修改玩法属性（各系统自行响应），它只做一件事：**在正确的时间呈现正确的选择，并将玩家的决定传达出去**。

## Player Fantasy

屏幕中央浮现三张卡片——利刃强化、风之步、铁壁。你的手停在键盘上方，快速扫了一眼：利刃强化能让你近战一刀更狠，但你这局已经在叠闪避了，风之步能让走位更丝滑，铁壁多 50 血容错更高。你选了风之步。卡片收起，战斗继续。三秒钟，一个决定，这一选可能决定你能不能撑过下两波。

## Detailed Design

### 1. 升级流程状态机

```
INACTIVE → SHOWING → SELECTING → CONFIRMED → INACTIVE
```

| 状态 | 说明 | 转换条件 |
|------|------|---------|
| INACTIVE | 等待波次系统触发 | 收到 `upgrade_window_requested` → SHOWING |
| SHOWING | 抽取升级并渲染界面 | 界面渲染完成 → SELECTING |
| SELECTING | 等待玩家选择 | 玩家选择一个升级 → CONFIRMED |
| CONFIRMED | 应用选择结果 | 应用完成 → 恢复游戏 → 调用 `upgrade_completed()` → INACTIVE |

### 2. 升级触发流程

1. 波次系统发出 `upgrade_window_requested` 信号
2. 本系统调用升级池的 `draw_upgrades(3)` 获取 3 个选项
3. 暂停游戏逻辑（敌人 AI、伤害、生成），UI 动画和输入保持运行
4. 渲染升级选择界面（3 张卡片，显示名称、描述、稀有度）
5. 等待玩家输入选择（键盘 1/2/3 或鼠标点击）
6. 玩家确认后调用升级池的 `acquire_upgrade(selected_id)`
7. 升级池发出 `upgrade_acquired` 信号，各玩法系统自行应用
8. 恢复游戏逻辑，调用波次系统的 `upgrade_completed()` 通知升级结束
8. 关闭界面，回到 INACTIVE

### 3. 升级选择界面布局

MVP 极简布局：屏幕中央横向排列 3 张卡片

```
┌─────────────┐ ┌─────────────┐ ┌─────────────┐
│  利刃强化    │ │   风之步     │ │    铁壁      │
│  COMMON      │ │  COMMON      │ │  COMMON      │
│             │ │             │ │             │
│ 近战伤害+20%│ │ 移速+15%    │ │ 生命+50     │
│             │ │             │ │             │
│   [1]       │ │   [2]       │ │   [3]       │
└─────────────┘ └─────────────┘ └─────────────┘
```

- 每张卡片显示：名称、稀有度标签、效果描述、快捷键提示
- 当前选中卡片高亮边框（键盘导航或鼠标悬停）
- 稀有度用边框颜色区分：COMMON=白色，UNCOMMON=绿色，RARE=蓝色

### 4. 输入处理

- **键盘**：1/2/3 数字键直接选择对应卡片，Enter 确认当前高亮
- **鼠标**：点击卡片直接选择并确认
- **导航**：左右方向键切换高亮卡片

### 5. 界面动画

- **进入**：3 张卡片从屏幕外滑入（0.3s），带轻微缩放弹性
- **选中**：选中卡片放大 + 发光，其余卡片缩小淡出（0.2s）
- **退出**：选中卡片向中心收束消失（0.2s），界面关闭

### 6. 无可选升级处理

- 如果 `draw_upgrades(3)` 返回空数组（所有升级已满层），跳过升级窗口
- 直接调用 `upgrade_completed()`，不显示界面
- 如果返回不足 3 个，按实际数量显示卡片

## Formulas

本系统无复杂公式，流程为纯事件驱动。仅记录界面时序参数：

```
# 界面动画时长
CARD_SLIDE_IN_DURATION = 0.3    # 卡片滑入动画（秒）
CARD_SELECT_DURATION = 0.2      # 选中动画（秒）
CARD_EXIT_DURATION = 0.2        # 退出动画（秒）

# 界面交互超时（与波次系统 upgrade_timeout 对齐）
UPGRADE_UI_TIMEOUT = 30.0       # 超时自动选择第一项（秒）
```

**超时自动选择**：玩家在 `UPGRADE_UI_TIMEOUT` 内未选择，自动选择高亮的第一张卡片，保证游戏不会卡死。

## Edge Cases

| 场景 | 预期行为 | 理由 |
|------|---------|------|
| `draw_upgrades()` 返回空数组 | 跳过界面，直接调用 `upgrade_completed()` | 无可选升级时不卡游戏流程 |
| `draw_upgrades()` 返回 1 个选项 | 显示单张卡片，玩家只有唯一选择 | 不足 3 个时按实际数量显示 |
| 超时自动选择 | 30s 后自动选择当前高亮卡片（默认第一张） | 与波次系统超时机制对齐，防止卡死 |
| 升级窗口期间游戏暂停 | 敌人 AI、伤害、生成全部冻结，UI 和输入正常运行 | 暂停让玩家安心选择，不牺牲交互响应 |
| `upgrade_window_requested` 在非 INACTIVE 状态收到 | 忽略，不重复触发 | 防止重复弹窗 |
| 玩家在动画播放中按键选择 | 缓存输入，动画结束后执行选择 | 输入不丢失，但不在动画中跳帧 |
| 同一帧内 `upgrade_completed()` 被调用两次 | 第二次忽略 | 防御性处理，防止波次系统重复触发 |

## Dependencies

| 依赖系统 | 方向 | 依赖内容 | 状态 |
|---------|------|---------|------|
| 波次系统 | 上游 → 本系统 | 发出 `upgrade_window_requested` 触发升级 | GDD 已通过 ✅ |
| 波次系统 | 本系统 → 上游 | 调用 `upgrade_completed()` 通知升级结束 | GDD 已通过 ✅ |
| 升级池/数据系统 | 本系统 → 下游 | 调用 `draw_upgrades()` 获取选项，`acquire_upgrade()` 确认选择 | GDD 已通过 ✅ |
| 升级池/数据系统 | 下游 → 本系统 | 发出 `upgrade_acquired` 信号（本系统不监听，各玩法系统监听） | GDD 已通过 ✅ |
| 生命/伤害系统 | 上游 | 升级窗口期间玩家可能死亡，需监听死亡信号 | GDD 已通过 ✅ |
| 输入系统 | 上游 | 玩家按键/点击选择升级 | GDD 已通过 ✅ |

**双向依赖说明**：
- 波次系统的 `upgrade_window_requested` / `upgrade_completed()` 交互协议已在双方 GDD 中确认一致
- 本系统不监听 `upgrade_acquired` 信号——效果应用是各玩法系统的职责，本系统只负责"选完传达"

## Tuning Knobs

| 旋钮 | 类型 | 默认值 | 安全范围 | 影响玩法 | 说明 |
|------|------|--------|---------|---------|------|
| `UPGRADE_UI_TIMEOUT` | float | 30.0 s | 10.0–60.0 s | 选择压力 | 超时自动选择，越短选择压力越大 |
| `CARD_SLIDE_IN_DURATION` | float | 0.3 s | 0.1–0.5 s | 节奏感 | 卡片入场动画时长 |
| `CARD_SELECT_DURATION` | float | 0.2 s | 0.1–0.4 s | 反馈感 | 选中动画时长 |
| `CARD_EXIT_DURATION` | float | 0.2 s | 0.1–0.3 s | 节奏感 | 退出动画时长 |

**配置方式**：动画时长定义在 `assets/data/upgrade_ui_config.tres`（Resource 文件）。

**备注**：MVP 动画全部为简单 Tween 实现，不依赖动画系统。超时值需与波次系统 `upgrade_timeout` 保持一致。

## Visual/Audio Requirements

**视觉**：
- 3 张卡片横向居中排列，带稀有度边框颜色区分（白/绿/蓝）
- 卡片入场：从屏幕下方滑入 + 缩放弹性
- 选中反馈：选中卡片放大发光，其余缩小淡出
- 背景半透明遮罩（alpha 0.5），游戏暂停但可看见当前战场状态

**音频**：
- 升级窗口打开时播放提示音——短促上升音调
- 卡片高亮切换时播放轻触音——细微的 UI 反馈
- 确认选择时播放确认音——满足感音效
- 音效资源路径：`assets/audio/sfx/upgrade_window_open.ogg`、`assets/audio/sfx/ui_hover.ogg`、`assets/audio/sfx/upgrade_confirm.ogg`

## UI Requirements

本系统是 MVP 中少数直接驱动 UI 的系统之一。

**升级选择界面**：
- 位置：屏幕中央
- 布局：3 张卡片横向等距排列，基于视口宽度自适应缩放（最小宽度保证文字可读）
- 每张卡片内容：名称、稀有度标签、效果描述文本、快捷键提示（1/2/3）
- 交互方式：键盘 1/2/3 直接选择，左右方向键导航 + Enter 确认，鼠标点击
- 高亮状态：选中卡片边框加粗发光

**已获取升级列表**：
- 位置：界面左下角
- 内容：竖排小字显示当前已获取的升级名称及层数（如"利刃强化 ×2"）
- 数据来源：从升级池系统读取 `acquired_upgrades` 字典

**无专属 HUD 元素**：本系统不负责常驻 HUD，只在升级窗口激活时显示选择界面。

## Acceptance Criteria

| 编号 | 标准 | 验证方式 |
|------|------|---------|
| AC-1 | 收到 `upgrade_window_requested` 后调用 `draw_upgrades(3)` | 单元测试：监听调用，断言参数正确 |
| AC-2 | 界面显示的卡片数量与 `draw_upgrades()` 返回数量一致 | 手动验证 / 集成测试 |
| AC-3 | 键盘 1/2/3 选择对应卡片并调用 `acquire_upgrade(id)` | 单元测试：模拟输入，断言调用参数匹配 |
| AC-4 | 鼠标点击卡片选择并调用 `acquire_upgrade(id)` | 手动验证 |
| AC-5 | 选择完成后调用波次系统 `upgrade_completed()` | 单元测试：断言调用顺序正确 |
| AC-6 | `draw_upgrades()` 返回空数组时跳过界面，直接调用 `upgrade_completed()` | 单元测试：断言无界面渲染 |
| AC-7 | 超时后自动选择第一张卡片 | 单元测试：等待超时，断言自动选择行为 |
| AC-8 | 升级窗口期间玩家死亡，界面立即关闭 | 单元测试：模拟死亡信号，断言状态回 INACTIVE |
| AC-9 | 非法状态下收到 `upgrade_window_requested` 不触发重复弹窗 | 单元测试：SELECTING 状态下发送信号，断言无重复调用 |
| AC-10 | 卡片显示正确的名称、描述、稀有度 | 手动验证 / 集成测试 |

## Open Questions

1. **升级窗口暂停的实现方式**：已决定暂停游戏。实现时需确保暂停不影响波次系统超时计时，且暂停仅冻结游戏逻辑（敌人、伤害），不冻结 UI 动画和输入。
2. **已获取升级列表布局细节**：已决定显示。具体位置和样式待 UI 实现时确定，初步方案为界面左下角竖排小字列表。
3. **卡片尺寸自适应方案**：已决定自适应。基于视口宽度按比例缩放卡片，最小宽度保证文字可读。
4. **卡片描述文本格式化**：当前 `UpgradeData.description` 为纯文本，是否需要支持富文本（如高亮数值）？MVP 纯文本即可。
